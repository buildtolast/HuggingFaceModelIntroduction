Why This Page Exists

The base guides cover what models exist and how they work. This page covers what you do with them in 2026: ground them in your data (RAG), let them take actions (agents, tool use), run them in production (deployment), and verify they're doing the right thing (evaluation, safety).

Most production LLM failures are not "the model isn't smart enough." They're: retrieval missed the right chunk, the tool schema was ambiguous, no guardrail caught the hallucination, or the deployment stack couldn't handle concurrent users. This page is about those problems.

1. Retrieval-Augmented Generation (RAG)

RAG is the dominant pattern for grounding LLMs in private or domain-specific data. Instead of fine-tuning on your corpus (expensive, opaque, risks catastrophic forgetting), you retrieve relevant chunks at query time and inject them into the prompt.

The Basic Pipeline

┌──────────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ User query │──▶│ Embed │──▶│ Vector │──▶│ Top-k │ │ "What is X?" │ │ (bge-*, │ │ search │ │ chunks │ └──────────────┘ │ e5, etc)│ │ (FAISS, │ └──────────┘ └──────────┘ │ pgvector│ │ │ Qdrant) │ ▼ └──────────┘ ┌──────────┐ │ Prompt │ │ + context│ └──────────┘ │ ▼ ┌──────────┐ │ LLM │──▶ Answer └──────────┘

Components & Current Best Practices (2026)

1. Chunking

  • Fixed-size (256-1024 tokens) with 10-20% overlap — simple default
  • Semantic/recursive chunking — splits on paragraph/section boundaries (LangChain RecursiveCharacterTextSplitter, Unstructured.io)
  • Late chunking (jina-embeddings-v3) — embed long doc first, then chunk the token embeddings. Preserves context.
  • Contextual retrieval (Anthropic, 2024) — prepend each chunk with an LLM-generated 1-line summary. Reduces retrieval failures 35-49%.

4. Hybrid Retrieval (Critical for Production)

Dense embeddings miss exact matches (SKU numbers, names, acronyms). Always combine:

  • Dense (semantic): bge-large / e5
  • Sparse (keyword): BM25, SPLADE
  • Fuse with Reciprocal Rank Fusion (RRF) or learned reranker

5. Reranking (the step most pipelines skip)

Retrieve top-50, rerank with a cross-encoder, keep top-5. Reranker sees query and candidate together, gives much better ordering.

cross-encoder/ms-marco-MiniLM-L-12-v2Very fastDecent baseline

Minimal RAG Example (Python)

from sentence_transformers import SentenceTransformer, CrossEncoder
from qdrant_client import QdrantClient
from transformers import pipeline

# 1. Embed corpus
embedder = SentenceTransformer("BAAI/bge-large-en-v1.5")
reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")

docs = [...]  # your chunks
vectors = embedder.encode(docs, normalize_embeddings=True)

# 2. Store
client = QdrantClient(":memory:")
client.upload_collection("kb", vectors=vectors, payload=[{"text": d} for d in docs])

# 3. Query
query = "What is RAG?"
q_vec = embedder.encode(query, normalize_embeddings=True)
hits = client.search("kb", q_vec, limit=50)

# 4. Rerank
pairs = [(query, h.payload["text"]) for h in hits]
scores = reranker.predict(pairs)
top_chunks = [h.payload["text"] for _, h in sorted(zip(scores, hits), reverse=True)[:5]]

# 5. Generate
llm = pipeline("text-generation", model="meta-llama/Llama-3.3-70B-Instruct")
context = "\n\n".join(top_chunks)
answer = llm(f"Context:\n{context}\n\nQuestion: {query}\nAnswer:")

RAG Frameworks

  • LlamaIndex — data framework, rich loader/parser ecosystem, advanced retrieval strategies (auto-merging, sentence-window, etc.)
  • LangChain — orchestration-heavy, popular, opinionated. Good for prototyping agents + RAG.
  • Haystack — production-focused, pipeline DAGs, strong eval tooling
  • DSPy (Stanford) — treats prompting + retrieval as optimizable programs; can compile prompts automatically
  • RAGatouille — makes ColBERT easy to use

Common RAG Failure Modes

Retrieval misses

Fix: hybrid search + reranking + better chunking.

Lost in the middle

Models attend weakly to mid-context. Fix: rerank to put best chunks at top/bottom.

Irrelevant retrieval

Top-k returns something even when nothing matches. Fix: threshold + "I don't know" prompt.

Conflicting chunks

Fix: deduplicate, use reranker to surface the most authoritative.

Chunk ≠ useful unit

Fix: include parent section header; use sentence-window retrieval.

Stale index

Re-embed periodically; invalidate on source change.

2. Agents: LLMs That Take Actions

An agent is an LLM wrapped in a loop that can call tools, observe results, and decide what to do next. The 2023 idea is now production-ready in 2026 thanks to better tool-use training (Llama 3+, Qwen2.5, Claude 3.5+, GPT-4+).

Agent Architectures

ReAct (Reason + Act)

The baseline pattern. Model alternates Thought → Action → Observation → Thought... until it has an answer.

Thought: I need the weather in Paris. Action: get_weather(city="Paris") Observation: 15°C, rainy Thought: I have enough to answer. Answer: It's 15°C and rainy in Paris.

Plan-and-Execute

Planner LLM writes a full plan first, executor runs steps. Better for complex tasks; worse for dynamic environments.

Reflexion / Self-Refine

Agent critiques its own output, tries again. Useful for code-gen, math, harder coding agents (SWE-agent, Cursor, Claude Code).

Multi-agent

Specialist agents (researcher, coder, reviewer) coordinate via messages. Frameworks: AutoGen, CrewAI, LangGraph. Useful in theory, expensive in practice — use only when single-agent has clearly failed.

Agent Failure Modes

  • Infinite loops — enforce max iterations (10-20 typical)
  • Tool selection errors — happens ~5-15% even for GPT-4. Mitigation: clear tool descriptions, fewer tools, reflection step
  • Hallucinated tool arguments — use structured output (JSON schema), validate before executing
  • Context explosion — long trajectories blow past context window. Mitigation: summarize old steps, sub-agent delegation
  • Cost explosion — one failed agent loop can cost $10-50 on commercial APIs. Set hard budgets and iteration caps.

3. Tool Use & Function Calling

Tool use lets a model request execution of a function (get_weather, search_database, send_email) and incorporate the result into its response. Native support exists in all modern models (GPT-4+, Claude 3+, Llama 3.1+, Qwen2.5+, Mistral Large+, Gemini).

Protocol (OpenAI/Anthropic format)

# 1. You describe tools as JSON schema
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Current weather for a city",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string"},
                "unit": {"type": "string", "enum": ["C", "F"]}
            },
            "required": ["city"]
        }
    }
}]

# 2. Model returns either a text answer OR a tool call
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "Weather in Tokyo?"}],
    tools=tools,
)

# 3. You execute, return the result as a tool_response message, loop
if response.choices[0].message.tool_calls:
    for call in response.choices[0].message.tool_calls:
        result = dispatch(call.function.name, json.loads(call.function.arguments))
        messages.append({"role": "tool", "tool_call_id": call.id, "content": result})
    response = client.chat.completions.create(model="gpt-4", messages=messages, tools=tools)

Best Practices

  • Fewer, better tools > many overlapping tools. 5-15 tools is the sweet spot for reliable selection.
  • Descriptive names & docstrings — the model picks based on these. search_internal_kb beats search.
  • Validate arguments — Pydantic or jsonschema before executing. Models occasionally hallucinate field names.
  • Idempotent tools when possible — retries are common.
  • Error messages back to the model — return errors as tool output so it can retry. Don't throw.
  • Parallel tool calls — GPT-4 and Claude 3.5+ can emit multiple calls per turn; batch execute.

Benchmark Context

On the Berkeley Function-Calling Leaderboard (BFCL) as of early 2026:

  • GPT-4, Claude 3.5+, Qwen2.5-72B: ~90%+ on simple tool calls
  • Llama 3.3 70B: ~88%
  • 7B models: 75-85% — usable with good prompts but require validation
  • Multi-turn / parallel / nested scenarios drop every model 10-20 points

4. Model Context Protocol (MCP)

MCP (introduced by Anthropic, late 2024) is an open standard for exposing tools, resources, and prompts to LLMs in a vendor-neutral way. Think of it as "LSP for LLM tools" — write a server once, any MCP-compatible client can use it.

Core Concepts

Tools

Functions the LLM can invoke (same as function calling). Defined with JSON schema.

Resources

Read-only data the LLM can reference (files, DB rows, API responses). Identified by URIs.

Prompts

Server-defined prompt templates users can invoke ("/summarize", "/code-review").

Transports

stdio (local process), Streamable HTTP (remote). JSON-RPC 2.0 underneath.

Why It Matters

  • Write once, run everywhere — an MCP server for GitHub works with Claude Desktop, Cursor, Zed, Windsurf, custom apps, etc.
  • Separates tool concerns — tool authors and app authors don't need to integrate pairwise
  • Ecosystem momentum — OpenAI, Google, Microsoft all announced MCP support in 2025

Minimal MCP Server (Python)

from mcp.server.fastmcp import FastMCP

app = FastMCP("weather-server")

@app.tool()
def get_weather(city: str, unit: str = "C") -> dict:
    """Get current weather for a city."""
    # ... your implementation
    return {"city": city, "temp": 15, "unit": unit, "conditions": "rainy"}

@app.resource("weather://cities")
def list_cities() -> str:
    """List supported cities."""
    return "Paris, Tokyo, New York, London"

if __name__ == "__main__":
    app.run()  # stdio transport by default

5. Reasoning Models & Test-Time Compute

The biggest shift of 2024-2026: models that think before answering. OpenAI's o1 (Sept 2024), DeepSeek-R1 (Jan 2025), and subsequent waves changed the frontier. These models generate a long hidden/visible chain-of-thought ("thinking"), then produce a concise answer.

Key Insight: Scale Test-Time Compute, Not Just Training Compute

Pre-2024: quality ≈ model size × training data. Post-o1: quality ≈ model size × training × inference-time thinking length. A 70B reasoning model can match a 400B non-reasoning model on math/code if you let it think.

Representative Models (2026)

Claude 3.7/Opus extended thinkingClosedVisible thinking traces, toggleable
DeepSeek-R1 / R1-Zero openOpen MITTrained with pure RL (GRPO). Thinking in <think> tags.
DeepSeek-R1-Distill-Qwen-{7B,14B,32B}OpenSmaller models distilled from R1 — punches above weight class
QwQ-32B (Qwen)Open ApacheStrong open reasoner, 32B
Qwen3 & Qwen3-ThinkingOpenToggleable reasoning mode
Gemini 2.5 ThinkingClosedGoogle's reasoning tier

How They're Trained (GRPO & Variants)

DeepSeek-R1-Zero demonstrated that pure RL, no supervised CoT data, can teach reasoning from scratch. The key algorithm:

  • GRPO (Group Relative Policy Optimization) — variant of PPO that samples a group of completions per prompt, computes relative rewards within the group, uses that as advantage. Cheap (no value model), effective for verifiable-reward tasks (math, code).
  • Reward functions: rule-based (correct answer? compiles? passes tests?) — not learned reward models, avoids reward hacking.
  • Emergent behavior: the model learns to backtrack, self-check, allocate more thinking to harder problems — purely from reward signal.

Inference-Time Techniques

  • Chain-of-Thought prompting — "Let's think step by step." (Works for non-reasoning models too)
  • Self-consistency — sample N solutions, majority-vote the answer
  • Tree-of-Thought — branch and evaluate partial reasoning paths
  • Best-of-N with verifier — sample N, score each with a reward/verifier model, pick best
  • Budget forcing (s1 paper, 2025) — force model to think longer by suppressing </think> token

When NOT to Use a Reasoning Model

  • Simple chat — reasoning is overhead (5-50x more tokens)
  • Latency-critical UX — thinking can take 10-60s
  • Cost-sensitive high-volume — reasoning tokens cost the same as regular output
  • Tasks that don't benefit: summarization, translation, casual Q&A
Rule of thumb: Use reasoning models for math, code with complex logic, multi-step planning, graduate-level science (GPQA). Use regular models for everything else.

6. Structured Output & Constrained Decoding

Getting an LLM to output valid JSON (or any other format) reliably is a solved problem in 2026. Three approaches, increasing reliability:

Approach 1: Prompt and Hope

"Respond in JSON with fields X, Y, Z." Works 90%+ on modern models but the 5-10% failure rate kills production pipelines.

Approach 2: Native Structured Output APIs

  • OpenAI Structured Outputs — pass a JSON schema, get guaranteed valid JSON (100%)
  • Anthropic tool use — tool arguments are schema-constrained
  • Gemini response_schema

Approach 3: Constrained Decoding (local models)

Mask out invalid tokens at each step. Guaranteed valid output, often faster than unconstrained because many tokens are "forced."

  • Replicate, Together, Anyscale, Fireworks, Groq — API providers for open models. Groq specializes in low-latency via LPUs (~500 tok/s on Llama 3).

  • Modal, RunPod, Lambda Labs — rent-a-GPU, bring your own server

Critical Optimizations

Continuous Batching

Don't wait for batch to fill. Stream new requests into running batch. 10-20x throughput.

PagedAttention

KV cache as pages, not contiguous. Near-zero fragmentation. Core to vLLM.

Prefix Caching

Cache KV for shared prompt prefixes (system prompts, few-shot). 5-10x speedup for chat.

Speculative Decoding

Draft model proposes N tokens, big model verifies in one pass. 2-3x latency win.

Quantization at Serve Time

AWQ/GPTQ 4-bit, FP8 (H100+). 4x memory, minor quality hit.

Tensor / Pipeline Parallelism

Shard model across GPUs. Needed for 70B+ on anything smaller than 80GB cards.

8. Evaluation: Did My Model/Pipeline Actually Work?

If you can't measure it, you're not engineering it. Evaluation is the gap that separates demos from production LLM systems.

Three Evaluation Layers

1. Static benchmarks (model-level)

  • MMLU / MMLU-Pro — knowledge and reasoning across subjects
  • GPQA (Diamond) — graduate-level science, hard
  • HumanEval / LiveCodeBench / SWE-Bench Verified — code
  • MATH / AIME — math reasoning
  • IFEval — instruction following
  • MT-Bench / AlpacaEval 2 / Arena-Hard — chat preferences
  • Chatbot Arena (LMSYS) — crowd-sourced Elo ratings; the most trusted single metric for general-purpose quality
  • BFCL — function/tool calling

Tooling: lm-evaluation-harness (EleutherAI), lighteval (HF), HELM (Stanford).

2. Task / application benchmarks (pipeline-level)

Static benchmarks don't tell you if your RAG pipeline works on your docs. Build your own.

  • Curate 50-500 representative queries with gold answers
  • Run your full pipeline, score with:
    • Exact match / F1 — for extraction/classification
    • ROUGE / BLEU — summarization (weak but cheap)
    • Retrieval metrics: Recall@k, MRR, nDCG — for RAG
    • LLM-as-judge — GPT-4 or Claude scores your output against a rubric
  • Track over time; regress before shipping.

3. Production monitoring (live)

  • Response latency (p50, p95, p99)
  • Token usage (cost)
  • User feedback (thumbs up/down, implicit signals)
  • Hallucination detection (fact-check against retrieved context)
  • Safety classifier alerts (guardrail trips)

Platforms: Langfuse (OSS, popular), Arize Phoenix (OSS), Braintrust, Weights & Biases Weave, LangSmith.

9. Safety, Guardrails & Alignment

Your model will occasionally: hallucinate, be jailbroken, leak PII, reproduce bias, produce unsafe content. Guardrails don't solve these — they catch and mitigate.

Threat Model

  • Prompt injection — user input overrides system instructions. Especially dangerous in agentic/tool-using systems.
  • Jailbreaks — getting the model to violate its alignment (DAN, many-shot, encoded prompts)
  • Hallucination — confident wrong answers
  • Data leakage — model regurgitates training data or context (PII, trade secrets)
  • Unsafe content — harmful, illegal, explicit
  • Bias / fairness — disparate performance or offensive output across groups

Frameworks

  • NVIDIA NeMo Guardrails — programmable rails via Colang DSL. Input / dialog / output / retrieval / execution rails.
  • Guardrails AI — "validators" for structure, PII, toxicity, competitors, etc.
  • LLM Guard — input/output scanners (prompt injection, bias, PII, secrets, toxicity)

Typical Production Stack

User input │ ▼ ┌────────────────────────┐ │ Input guardrails │ ── Llama Guard 3, prompt-injection classifier │ - topic allow-list │ │ - PII redaction │ └────────────────────────┘ │ ▼ ┌────────────────────────┐ │ LLM + RAG + Tools │ └────────────────────────┘ │ ▼ ┌────────────────────────┐ │ Output guardrails │ ── Llama Guard 3, PII scan, hallucination check │ - fact grounding │ (answer in retrieved context?) │ - safety classification│ │ - JSON schema validate │ └────────────────────────┘ │ ▼ Response to user (or refusal)

Hallucination Mitigation Checklist

  • Ground in retrieved context (RAG) and prompt "Only answer from the context"
  • Instruct to cite sources; reject answers without citations
  • Post-hoc fact-checking: NLI model checks each claim against context
  • Lower temperature for factual tasks (0.0-0.3)
  • Use a reasoning model for complex chains
  • Calibrate via "I don't know" examples in prompts/fine-tuning data

10. Prompt Engineering (What Still Matters in 2026)

Modern models are less brittle to prompt wording than 2023 models — but prompt engineering still delivers the biggest wins per hour invested. The techniques that consistently work:

High-Leverage Techniques

System message + role clarity

Dedicated system message. "You are a [role]. Your job is [goal]. Constraints: [rules]."

Few-shot with diverse examples

2-5 examples covering edge cases. Huge impact on format and tone.

XML / Markdown structure

Claude loves <context>...</context>. GPT likes markdown headings. Use what the model was trained on.

Chain of Thought

"Think step by step before answering." Still works even on reasoning models (redundant for o1/R1).

Output format specification

Exact JSON schema or response template. Combine with structured output APIs.

Explicit negative constraints

"Do NOT invent facts not in the context." "Do NOT answer if unsure."

Context ordering

Most important info at start or end (lost-in-middle problem).

Anchor with a concrete example

Instead of abstract rules: show one full input → output pair.

Model-Family Quirks

  • Claude — strong system prompts, loves XML tags, use "Assistant:" prefill to shape output
  • GPT-4 — responds well to markdown, detailed multi-section system prompts
  • Llama — respect the chat template exactly (system/user/assistant tokens); tokenizer mismatches kill output quality
  • Qwen — bilingual prompts OK; strong at JSON
  • Reasoning models (o1/R1) — less prompt engineering needed, especially no need for "think step by step"; simpler prompts often better

Tools

  • DSPy — compile prompts automatically from examples; treats prompting as program optimization
  • promptfoo — eval prompt variants systematically
  • LangSmith / Langfuse — version prompts, A/B test in production
