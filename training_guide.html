Training Fundamentals

What is Fine-tuning?

Fine-tuning adapts a pre-trained model to your specific task by training it on your own data. The model retains general knowledge while learning task-specific patterns.

Pre-trained Model

Trained on 2+ trillion tokens of internet data. General knowledge but not specific to your domain.

Fine-tuning

Further training on 1K-100K examples of your specific task. Adapts the model to your use case.

Result

Task-specific model with better accuracy, lower latency, and smaller size than prompt engineering.

Beginner: Simple Fine-tuning (20-line Code)

BEGINNER
Goal: Get a working fine-tuning pipeline running. We'll use Google Colab (free GPU) and Hugging Face Trainer (easiest API).

Step 1: Setup Environment

# Install required packages !pip install transformers datasets peft bitsandbytes -q # Login to Hugging Face from huggingface_hub import login login() # You'll be prompted for your HF token

Step 2: Minimal Fine-tuning Example

import torch from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer from datasets import load_dataset # 1. Load model and tokenizer # Use FP16 to halve memory — without this, a 7B model loads in FP32 and OOMs on Colab T4. model_name = "meta-llama/Llama-3.1-8B" model = AutoModelForCausalLM.from_pretrained( model_name, torch_dtype=torch.float16, device_map="auto", ) tokenizer = AutoTokenizer.from_pretrained(model_name) # 2. Load your data dataset = load_dataset("text", data_files={"train": "train.txt"}) # 3. Tokenize def tokenize(sample): return tokenizer(sample["text"], truncation=True, max_length=512) dataset = dataset.map(tokenize, batched=True) # 4. Setup training args = TrainingArguments( output_dir="my-fine-tuned-model", num_train_epochs=3, per_device_train_batch_size=4, learning_rate=2e-4, ) # 5. Train trainer = Trainer(model=model, args=args, train_dataset=dataset["train"]) trainer.train() # 6. Save model.save_pretrained("my-fine-tuned-model")

Expected Results

  • Training time: 2-4 hours on Colab (free T4 GPU)
  • VRAM needed: ~24GB (use FP16 or quantization)
  • Model size: 14GB (7B params × 2 bytes for FP16)
  • Quality improvement: 10-30% on your specific task
Common Issues:
  • Out of Memory? Reduce batch_size to 2 or use quantization
  • Model not loading? Ensure you have access to the model (e.g., accept Llama license)
  • Slow training? Normal on free GPU. Use paid GPU for 10× speedup

Intermediate: LoRA (Parameter-Efficient Fine-tuning)

INTERMEDIATE
Why LoRA? Train only 1.6M parameters (0.02%) instead of 7B. 75% VRAM savings, faster training, same quality.

LoRA Architecture

Instead of updating W (7B params): W_new = W + ΔW With LoRA, decompose ΔW: ΔW = B @ A (where A: 7B×8, B: 8×7B) ↑ Only train these (1.6M params) Result: Same expressiveness, 1000× fewer parameters!

Complete LoRA Example

from peft import LoraConfig, get_peft_model from transformers import AutoModelForCausalLM, AutoTokenizer, Trainer, TrainingArguments from datasets import load_dataset # 1. Load model model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.1-8B") tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.1-8B") # 2. Configure LoRA lora_config = LoraConfig( r=8, # Rank lora_alpha=16, # Alpha for scaling target_modules=["q_proj", "v_proj"], # Which layers to adapt lora_dropout=0.05, bias="none", task_type="CAUSAL_LM" ) # 3. Apply LoRA model = get_peft_model(model, lora_config) model.print_trainable_parameters() # Output: trainable params: 1,638,400 || all params: 6,738,415,616 (0.02%) # 4. Prepare data dataset = load_dataset("text", data_files={"train": "train.txt"}) def preprocess(sample): return tokenizer(sample["text"], truncation=True, max_length=512) dataset = dataset.map(preprocess, batched=True) # 5. Train training_args = TrainingArguments( output_dir="lora-model", per_device_train_batch_size=16, # Can use larger batch with LoRA num_train_epochs=3, learning_rate=2e-4, save_strategy="epoch", logging_steps=100, ) trainer = Trainer( model=model, args=training_args, train_dataset=dataset["train"], ) trainer.train() # 6. Save only LoRA weights (~50MB!) model.save_pretrained("lora-adapter")

LoRA Best Practices

Parameter Typical Value What it does
r (rank) 8-16 Higher = more expressive but slower. 8 is usually enough
lora_alpha 16-32 Scaling factor. Usually 2× r
target_modules q_proj, v_proj Which transformer layers to adapt. Start with Q and V
lora_dropout 0.05-0.1 Regularization. Prevents overfitting

Using a LoRA-trained Model

from peft import AutoPeftModelForCausalLM # Load base model + LoRA adapter model = AutoPeftModelForCausalLM.from_pretrained("lora-adapter") # Use like normal inputs = tokenizer("Hello,", return_tensors="pt") outputs = model.generate(**inputs) print(tokenizer.decode(outputs[0]))

Advanced: QLoRA (Quantized LoRA)

ADVANCED
QLoRA = LoRA + 4-bit Quantization
Run 7B models on 8GB VRAM (RTX 4090). Same quality as LoRA, 3× cheaper hardware.

QLoRA Complete Example

from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training # 1. Quantization config (4-bit) bnb_config = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype="float16", bnb_4bit_use_double_quant=True, ) # 2. Load model in 4-bit model = AutoModelForCausalLM.from_pretrained( "meta-llama/Llama-3.1-8B", quantization_config=bnb_config, device_map="auto" ) # 3. Prepare for training model = prepare_model_for_kbit_training(model) # 4. Setup LoRA on quantized model lora_config = LoraConfig( r=8, lora_alpha=16, target_modules=["q_proj", "v_proj"], lora_dropout=0.05, bias="none", task_type="CAUSAL_LM" ) model = get_peft_model(model, lora_config) # 5. Training (same as LoRA) # ... (see LoRA example above)

QLoRA Memory Breakdown

Component 7B Model Note
Model weights (4-bit NF4) ~3.9GB 7B × 4 bits / 8 = 3.5GB + NF4 scale/zero-point overhead
LoRA adapters ~0.1GB 1.6M params × FP32
Optimizer states 2-4GB 8-bit Adam
Batch data + cache 2-3GB batch_size=4, seq_len=512
TOTAL 6-9GB Fits in RTX 4090 (24GB)

Advanced Techniques

1. Multi-GPU Training with DeepSpeed

# For 8+ GPUs, use DeepSpeed for distributed training training_args = TrainingArguments( output_dir="model", num_train_epochs=3, per_device_train_batch_size=8, gradient_accumulation_steps=4, optim="adafactors", # Memory efficient optimizer deepspeed="ds_config.json" # DeepSpeed config ) # ds_config.json (minimal working config): # Note: train_batch_size must equal # train_micro_batch_size_per_gpu * gradient_accumulation_steps * world_size # Below assumes 8 GPUs * micro-batch 8 * accum 1 = 64 { "train_batch_size": 64, "train_micro_batch_size_per_gpu": 8, "gradient_accumulation_steps": 1, "fp16": {"enabled": true}, "optimizer": {"type": "Adam", "params": {"lr": 0.0002}}, "zero_optimization": { "stage": 2, "offload_optimizer": {"device": "cpu"} } }

2. DPO (Direct Preference Optimization)

Train with human preference data instead of RLHF:

from trl import DPOTrainer # Format: (prompt, chosen_response, rejected_response) # Example: # "What is 2+2?" -> "4" (chosen) vs "5" (rejected) trainer = DPOTrainer( model=model, args=training_args, train_dataset=preference_dataset, beta=0.1, # Temperature for preference ) trainer.train()

3. Custom Training Loop with Accelerate

from accelerate import Accelerator accelerator = Accelerator() model, optimizer, train_dataloader = accelerator.prepare( model, optimizer, train_dataloader ) for epoch in range(num_epochs): for batch in train_dataloader: outputs = model(**batch) loss = outputs.loss accelerator.backward(loss) optimizer.step() optimizer.zero_grad()

5 Pitfalls That Kill Fine-tuning Projects

Most failed fine-tunes fail for the same handful of reasons. Read these before you start training, not after.

1. Catastrophic Forgetting

What it looks like: Your fine-tuned model aces your task but has forgotten how to do basic things it could do before — simple Q&A, small talk, reasoning.

Why it happens: You trained for too many epochs on a narrow dataset. The optimizer overwrote general-purpose weights with task-specific patterns.

Fix:

  • Use LoRA/QLoRA — adapters isolate changes and leave base weights frozen
  • Stop at 1–3 epochs (not 10); monitor validation loss and stop when it plateaus
  • Mix in 10–20% general instruction data (e.g., a slice of OpenAssistant or Dolly)
  • Use a low learning rate (1e-4 to 2e-4 for LoRA, 1e-5 to 5e-5 for full FT)

2. Overfitting on a Tiny Dataset

What it looks like: Training loss drops to ~0, but on prompts even slightly different from your training data the model produces garbage or memorized snippets.

Why it happens: <1K examples plus many epochs = the model memorizes rather than generalizes.

Fix:

  • Hold out 10–20% of data as a validation set. Stop when val loss stops improving.
  • If you have <1K examples, don't fine-tune — use few-shot prompting instead
  • Increase lora_dropout to 0.1 and add weight_decay=0.01
  • Augment your data: paraphrase, vary phrasing, add negative examples

3. Learning Rate Disasters

What it looks like: Loss explodes to NaN after a few steps, or loss plateaus immediately and never drops.

Why it happens: LR too high → gradient updates blow up and corrupt the model. LR too low → no meaningful learning happens.

Fix:

  • Default starting points: LoRA 2e-4, QLoRA 2e-4, full FT 2e-5
  • Always use a warmup: warmup_ratio=0.03 in TrainingArguments
  • Use a cosine or linear schedule: lr_scheduler_type="cosine"
  • If loss hits NaN, halve the LR and try again — do not continue from a corrupted checkpoint
  • Clip gradients: max_grad_norm=1.0

4. Wrong Chat Template / Token Format

What it looks like: Model trained fine (loss dropped), but at inference it produces gibberish, repeats special tokens, or ignores your instructions.

Why it happens: Each model family (Llama 3, Qwen, Mistral) uses different chat templates and special tokens. Training with the wrong template — or raw text when the base expects a template — breaks generation.

Fix:

  • Use tokenizer.apply_chat_template(messages, tokenize=False) — never hand-roll the format
  • Train on the Instruct variant if you want chat behavior; train on the base variant only for raw text completion
  • Verify by tokenizing a sample and visually inspecting the special tokens (<|im_start|>, [INST], etc.) are where you expect
  • Mask loss on the prompt — only compute loss on the response tokens (use DataCollatorForCompletionOnlyLM from TRL)

5. Evaluating Only on Training-Like Data

What it looks like: Your eval metrics look great. You deploy. Users immediately find failure modes you never saw.

Why it happens: Your eval set was drawn from the same distribution as training, or is too small to catch tail failures.

Fix:

  • Build a held-out adversarial eval set: prompts phrased unusually, edge cases, out-of-scope questions
  • Check for data contamination: search your training data for eval questions (literal substring match)
  • Run a side-by-side comparison against the base model on generic prompts — confirm you didn't regress general abilities
  • For chat models, evaluate qualitatively on 50 real user prompts — quantitative metrics miss tone and helpfulness

Troubleshooting Guide

Out of Memory (OOM)

Error: RuntimeError: CUDA out of memory
  1. Reduce batch_size

    per_device_train_batch_size = 2 (try smaller)

  2. Reduce sequence length

    max_length = 256 instead of 512

  3. Use gradient accumulation

    gradient_accumulation_steps = 4 (simulates batch_size × 4)

  4. Enable gradient checkpointing

    model.gradient_checkpointing_enable()

  5. Use 8-bit or 4-bit quantization

    Reduces model VRAM by 50-75%

Model Not Converging

Issue Solution
Loss doesn't decrease Lower learning_rate (try 1e-5) or increase training epochs
Loss spikes randomly Gradient clipping: max_grad_norm = 1.0
Loss oscillates Learning rate too high. Try 0.5× current value
Overfitting (train loss ↓, val loss ↑) Add more training data, increase dropout, reduce epochs

Training is Slow

  • Check GPU utilization: nvidia-smi should show 80%+ usage
  • Increase batch size (if VRAM allows)
  • Use mixed precision: fp16=True
  • Enable Flash Attention: Much faster attention mechanism
  • Use faster GPU (H100 >> A100 >> V100)

Model Quality Issues

Problem Cause Fix
Output is gibberish Data format wrong or tokenizer issue Verify data format, check tokenization
Model ignores your instructions Too little instruction-following data Add more diverse instruction examples
Repetitive outputs Training data has repetitive patterns Clean data, add diversity, increase top_p
